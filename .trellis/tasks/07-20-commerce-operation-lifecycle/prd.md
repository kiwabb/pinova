# 完善商城交易与运营闭环

## Goal

在现有 Pinova 商城购买主干之上，完成订单履约、用户取消、退款售后，以及商品、SKU、库存、会员、类目的后台运营能力，使订单从创建到支付、履约、完成、关闭或售后都存在真实可操作的业务闭环。

## What I Already Know

* C 端已经具备注册登录、商品浏览、购物车、地址、下单、Mock 支付和会员订单列表。
* 下单事务已经实现 checkout 幂等、按店铺拆单、成交快照、库存预占、零元订单直接扣减，以及购物车消费。
* 支付成功会把库存预占转为实扣；支付超时会释放预占并关闭待支付订单。
* 交易订单已有 `PENDING_PAYMENT / PENDING_FULFILLMENT / FULFILLING / COMPLETED / CLOSED` 状态，但当前没有取消、发货、完成的业务写接口。
* 当前只有本地 Mock 支付 Provider，没有真实退款能力；因此退款领域必须先定义稳定接口与状态机，首期可由 Mock Provider 完成可测试闭环。
* 后台已有独立 React + Vite + Ant Design 应用、管理员认证、`ORDER_READ` 权限、订单列表和订单详情。
* 后台会员、类目页面仍是空状态；商品、SKU、仓库、库存 Controller 是空壳。
* 数据库已有会员、类目、SPU、SKU、仓库、库存、订单、支付等基础表，但尚无退款单、售后单和订单履约操作记录表。
* 当前端到端测试主要 mock HTTP，交易核心缺少 PostgreSQL 集成与并发测试。

## Assumptions (Temporary)

* 本任务以单体仓库内的完整 MVP 为目标，不接入真实第三方支付、物流承运商或短信服务。
* 实物订单首期采用“整单发货、整单确认完成”，不在第一版实现一单多包裹。
* 售后首期只支持仅退款，不包含退货物流、换货或维修。
* 后台写操作继续沿用管理员会话、CSRF、权限码和审计字段约束。
* 商品与库存后台必须操作真实 API 和数据库，不提供假数据或仅展示的占位页面。

## Open Questions

* 无。

## Requirements (Evolving)

* 后台可以把待履约订单推进为履约中，再推进为已完成，并记录操作人和时间。
* 履约中订单支持三种完成入口：会员确认收货、到期调度自动完成、后台填写原因强制完成；三者复用同一状态迁移服务并记录完成来源。
* 自动完成期限固定为发货后 7 天；订单进入履约中时写入 `autoCompleteAt`，调度任务只处理已到期且仍为履约中的订单。
* 用户可以在规则允许时取消订单；取消必须与支付结果、超时任务和库存处理保持并发一致。
* 待支付取消以整个 checkout 为粒度：取消任一入口都会关闭该 checkout 的全部待支付子订单和待支付/失败支付单，并释放全部库存预占。
* 已支付订单不能走取消接口，只能按单笔子订单进入仅退款售后。
* 建立退款与售后领域模型，提供会员端申请/查询和后台审核/处理入口。
* 售后首期只支持仅退款：未发货和已发货订单均不建立退货物流；退款成功不自动回补库存。
* 仅退款以按店铺拆出的单笔 `trade_order` 为粒度，只允许整单全额退款；不支持部分金额、部分商品、数量退款或同一订单拆成多次退款。
* 仅退款从支付成功后即可申请；待履约、履约中均可申请，已完成订单在完成后 7 天内可申请，超过服务端截止时间后拒绝。
* 售后处于审核中或退款中时冻结发货、会员确认收货、自动完成和后台强制完成；售后拒绝后恢复推进资格，退款成功后订单进入新增 `REFUNDED` 状态。
* 后台提供真实的商品 SPU、SKU、类目、仓库库存、会员查询和维护能力。
* 所有写接口必须有权限、请求校验、稳定错误码、幂等或版本冲突保护和审计记录。
* 所有状态变更必须由服务层状态机校验，禁止 Controller 或前端直接写状态字段。
* 首期后台写操作只允许 `SUPER_ADMIN`；其他管理员保留既有读权限但不能执行订单、售后、商品、库存、会员或类目写操作。
* 会员后台提供真实列表和详情，并允许 `SUPER_ADMIN` 启用/禁用会员以及强制全部登录会话下线；不支持重置密码或直接编辑手机号、邮箱。
* 禁用会员时必须同步使其全部有效会话失效，后续登录请求被拒绝；重新启用不恢复旧会话。
* 类目后台支持新增、编辑、移动、同级排序和软删除；仅没有子类目且没有关联商品的叶子类目允许删除。
* 类目写操作携带 `version`；移动必须阻止循环引用并遵守现有层级约束。
* 商品后台首期只允许创建和销售实物商品；不开放虚拟商品或服务商品的录入与上架。
* SPU/SKU 后台支持真实列表、详情、新增、编辑、启用/停用、上架/下架和软删除；上架前校验资料完整、至少一个启用 SKU 和合法库存模式。
* 库存后台同时支持入库/出库增量调整和盘点修正；每次调整必须填写原因、业务幂等键并携带 `version`，同步写不可变库存流水。
* 库存调整不得使 `onHandQuantity < reservedQuantity`，后台不得直接修改预占数量。
* 后台提供仓库列表、新增、编辑和启用/停用；存在活动预占的仓库不能停用。
* 实物订单采用整单发货；发货必须填写承运商和物流单号，首期仅保存并展示物流信息，不接第三方轨迹 API。
* 发货命令必须幂等；物流信息更正使用独立后台命令并保留原值、新值、原因和操作人审计。
* 商品后台完整包含 SPU 基础资料、结构化详情区块、商品图库、主图、SKU 专属图片和 MinIO 文件上传。
* 商品详情只允许现有白名单区块；媒体上传限制 MIME、大小和对象 Key 命名，删除或替换媒体时处理未引用对象。
* 建立统一后台操作审计记录和查询页面，覆盖订单、售后、退款、商品、SKU、媒体、库存、仓库、会员和类目写操作。
* 退款失败支持 `SUPER_ADMIN` 人工重试和渠道结果对账；必须复用原退款单和业务幂等键，禁止创建第二笔退款。

## Acceptance Criteria (Evolving)

* [ ] 已支付订单可由后台开始履约并完成，非法状态跳转被拒绝。
* [ ] 履约中订单可由会员确认收货、到期任务自动完成或后台强制完成；并发执行时只有一次状态迁移成功。
* [ ] 后台强制完成必须填写原因，所有完成入口均记录来源、操作人和时间。
* [ ] 自动完成时间严格等于发货时间加 7 天，重复扫描已完成订单不会产生第二次状态流水。
* [ ] 待支付订单可取消并释放预占；支付成功竞态下不会同时关闭与扣库存。
* [ ] checkout 取消与支付成功、支付超时并发执行时只有一个方向提交；重复取消幂等返回最终关闭结果。
* [ ] 符合规则的已支付订单可发起仅退款售后，后台可审核并执行 Mock 退款。
* [ ] 退款金额不会超过实付金额，多次请求不会重复退款。
* [ ] 一笔子订单最多形成一笔有效仅退款售后和一笔全额退款单；重复申请或重复执行返回原结果或稳定冲突。
* [ ] 完成订单的售后截止时间严格等于完成时间加 7 天，边界时间由服务端统一判断。
* [ ] 活动售后存在时所有履约推进入口返回稳定冲突；拒绝后可继续履约，退款成功后任何履约入口均不可再推进。
* [ ] 仅退款完成后不修改已扣减库存，不生成退货入库流水。
* [ ] 商品、SKU、类目和库存后台具备真实列表、详情及约定的写操作。
* [ ] 会员后台具备真实列表和详情，敏感字段按权限与脱敏规则输出。
* [ ] 关键状态机、金额、库存补偿、幂等和并发路径有数据库集成测试。
* [ ] C 端与后台端的主要成功、空、错误、权限和冲突状态可操作。
* [ ] 非 `SUPER_ADMIN` 调用任一后台写接口均返回稳定的 403 错误，前端同时不展示写操作入口。
* [ ] 禁用会员后所有旧会话立即失效且不能登录；重新启用后必须重新登录。
* [ ] 有子类目或关联商品的类目删除被拒绝；并发编辑、移动和排序不会静默覆盖。
* [ ] 非实物商品创建被拒绝；不满足完整性条件的 SPU 不能上架。
* [ ] 重复库存调整不会重复入账，盘点和出库不能把现货调整到低于预占数量。
* [ ] 未填写承运商或物流单号不能发货；重复发货不产生第二条履约记录。
* [ ] 运营人员可从后台完成商品资料、详情、图库、主图和 SKU 图片维护，前台读取后正确展示。
* [ ] 后台可以按操作者、领域、动作、目标和时间查询审计记录，敏感凭据和渠道原文不进入审计详情。
* [ ] 失败退款可安全重试和主动对账，重复操作不会造成重复退款。

## Research Notes

### Feasible Approaches

**Approach A: 显式状态机 + 独立业务表（Recommended）**

* 新增履约记录、售后单、退款单，扩展支付 Provider 退款接口。
* 复用现有 checkout 锁、行锁、version 和库存流水模式。
* 边界清楚，支持审计、重试和未来真实渠道。

**Approach B: 直接扩展订单/支付表**

* 文件较少，但无法稳健支持多次售后、退款重试和完整审计。

**Approach C: 事件驱动 Saga**

* 扩展性高，但当前缺少消息、Outbox、重放和监控基础设施，首期风险过高。

## Decision (ADR-lite)

**Context**: 本任务需要在现有模块化单体中补齐多个跨域写流程，同时控制首期复杂度。

**Decision**: 采用显式状态机 + 独立履约/售后/退款单据 + 同库事务补偿。售后首期只支持仅退款；订单支持用户确认、发货后 7 天自动完成和后台强制完成；后台写操作仅限 `SUPER_ADMIN`。

**Consequences**: 保留清晰审计和未来真实支付扩展点；首期不处理退货物流与普通角色写权限配置。退款失败通过人工重试和主动对账闭环，不引入消息基础设施。

## Technical Approach

* 新增前向数据库迁移：订单履约字段/记录、售后单、退款单、状态流水、后台审计；扩展管理员超级角色校验。
* 交易域新增显式命令服务：checkout 取消、整单发货、物流更正、用户确认、后台强制完成、自动完成。
* 售后域新增申请、审核、拒绝、退款执行、失败重试和主动对账状态机；扩展 Mock Payment Provider 的退款协议。
* 库存域继续使用行锁、version 和不可变流水；仅待支付 checkout 取消释放预占，仅退款不回补已扣库存。
* 后台按 feature 增量实现会员、类目、商品、SKU、媒体、仓库、库存、售后和审计；所有写入口只对 SUPER_ADMIN 可见且服务端强校验。
* C 端增加取消 checkout、订单物流、确认收货、售后申请与进度查询。
* 使用 PostgreSQL 集成测试覆盖交易竞态，并增加真实 API 浏览器冒烟流程；不以 HTTP mock 作为交易闭环的唯一证据。

## Implementation Plan

1. 数据库迁移、状态机契约、超级管理员写权限和统一审计基础设施。
2. checkout 取消、整单发货、物流更正、三种完成入口及 C 端/后台交互。
3. 仅退款售后、Mock 退款 Provider、审核、重试、对账及两端页面。
4. 会员与类目真实后台管理。
5. SPU、结构化详情、媒体上传、SKU 完整后台管理。
6. 仓库、库存查询、增量调整、盘点修正与流水后台。
7. PostgreSQL 并发集成测试、真实 API 浏览器冒烟、文档和 Obsidian 流程图更新。
