# 交易与运营闭环领域设计研究

## 现有约束

* 工程是模块化单体，交易、支付、库存共用 PostgreSQL，可以使用同库本地事务。
* `checkoutNo` advisory lock 已经串行化下单、支付结果和超时关闭。
* 库存余额采用行锁 + version 条件更新，库存流水不可变。
* 支付渠道已有 Provider 抽象，但只支持创建、查询、关闭，不支持退款。
* 后台已有独立身份、角色、权限和 CSRF 基础设施，可以扩展领域权限。
* 订单表状态已包含待履约、履约中、完成、关闭，但缺少显式状态迁移服务和履约记录。

## 可行方案

### A. 显式领域状态机 + 独立业务表（推荐）

* 订单状态继续表示主生命周期。
* 新增履约操作记录、退款单、售后单及状态流水。
* 扩展 `PaymentProvider` 的退款能力，Mock Provider 提供本地闭环。
* 取消、退款、履约操作均通过专门服务完成，并复用 checkout/order 行锁和库存 version 保护。
* 后台管理按领域提供查询/命令 API，权限码细分为 READ/WRITE。

优点：边界清楚、历史可审计、支持重试和将来接真实渠道；与现有模块化单体一致。

代价：迁移、实体、Mapper、服务、API、C 端和后台改动较多，需要拆成多个可验证子任务。

### B. 直接扩展 trade_order/payment_order

* 在现有表增加退款金额、退款状态、物流字段和售后原因。
* 所有动作集中进订单服务。

优点：初始文件少。

缺点：一笔订单存在多次售后或退款重试时无法正确建模；支付、售后、履约职责耦合；审计和幂等困难。不推荐。

### C. 事件驱动 Saga

* 交易、库存、支付、售后通过领域事件和补偿动作协作。

优点：适合未来拆服务和高吞吐。

缺点：当前没有消息基础设施、Outbox、事件监控和重放工具；为了本次需求引入会放大交付风险。不推荐首期采用。

## 推荐状态边界

* 主订单：待支付 → 待履约 → 履约中 → 已完成；待支付可关闭。
* 履约：首期整单履约，记录承运信息、发货时间、完成时间和操作人。
* 售后：申请中 → 已同意/已拒绝 → 退款中 → 已完成/失败/已关闭。
* 退款：独立幂等单据，金额累计不得超过订单实付金额；Provider 结果与本地推进分离。
* 取消：待支付释放预占并关闭；已支付不直接取消，转售后退款流程。

## 并发与幂等

* 订单命令优先锁订单；涉及同 checkout 的支付结果或超时竞争时，先取得 checkout advisory lock，再按既定顺序锁支付、订单、预占和库存。
* 后台写操作携带 version，受影响行数不是 1 时返回稳定冲突码。
* 退款使用业务幂等键和渠道退款号唯一索引，重复提交返回原退款单。
* 售后审核、退款回调和人工重试必须具有状态前置条件，禁止重复退款。

## 后台运营边界

* 权限建议细分：`ORDER_READ/WRITE`、`AFTER_SALE_READ/WRITE`、`PRODUCT_READ/WRITE`、`INVENTORY_READ/WRITE`、`MEMBER_READ/WRITE`、`CATEGORY_READ/WRITE`。
* SUPER_ADMIN 自动持有全部内置权限；普通角色通过现有关系表授权。
* 商品、SKU、类目采用软删除和 version；库存只允许通过调整命令写余额并同步写不可变流水。
* 会员首期允许查询、启用/禁用和会话失效，不允许后台读取密码摘要或直接修改认证凭据。

## 测试重点

* PostgreSQL 真实事务：支付成功 vs 超时、取消 vs 支付成功、重复退款、库存调整 vs 下单。
* API 权限和 CSRF：读写权限分离。
* 前端：C 端申请/取消与后台履约/审核的真实 API 流程，不只 mock HTTP。
